Namens- und Ordner-Matching

Dokumentation der Zuordnungslogik im Employee Document Matcher

Kernaussage: Das System trifft keine freie KI-Entscheidung über Personen. Es sucht deterministisch nach vollständigen Namen bekannter Mitarbeitender im extrahierten PDF-Text. Die Ordnerzuordnung ist fachlich eine Kategoriezuordnung und wird erst nach manueller Bestätigung verbindlich gespeichert.

1. Ausgangspunkt nach dem PDF-Upload

Wenn ein PDF hochgeladen wird, speichert das Backend die Datei zunächst unter einem technischen Speicherschlüssel. Dieser Speicherschlüssel ist eine zufällige UUID mit der Endung .pdf. Der ursprüngliche Dateiname wird zusätzlich in der Datenbank gespeichert, hat aber keinen Einfluss auf die automatische Zuordnung.

Danach wird der Text aus dem PDF extrahiert. Nur wenn die Textextraktion erfolgreich ist, laufen Namensmatching und Kategorie-Matching. Ist das PDF leer, verschlüsselt, beschädigt oder rein bildbasiert, kann der Inhalt nicht zuverlässig geprüft werden. In diesem Fall erhält die Analyse den Status UNREADABLE und landet im Review.

2. Namensmatching

Das Namensmatching findet in der Klasse PersonMatcher statt. Es arbeitet mit einer kontrollierten Kandidatenliste: Alle Mitarbeitenden werden aus der Datenbank gelesen, und aus Vorname und Nachname wird jeweils ein vollständiger Name gebildet, zum Beispiel Anna Müller.

  1. PDF-Text wird aus dem Dokument extrahiert.
  2. Alle Mitarbeitenden werden aus der Datenbank geladen.
  3. Für jeden Mitarbeitenden wird ein Kandidat aus employeeId und vollständigem Namen gebaut.
  4. PDF-Text und Kandidatenname werden normalisiert.
  5. Für jeden vollständigen Namen wird geprüft, ob er als eigenständiger Treffer im Text vorkommt.
  6. Aus der Anzahl der Treffer entsteht der Status MATCHED, NO_MATCH oder AMBIGUOUS.

Normalisierung

Vor dem Vergleich normalisiert das System sowohl den PDF-Text als auch den Kandidatennamen:

  • Unicode-Normalisierung mit NFKC
  • Umwandlung in Kleinschreibung
  • Entfernen führender und nachfolgender Leerzeichen
  • Zusammenfassen mehrerer Leerzeichen, Tabs oder Zeilenumbrüche zu einem Leerzeichen

Dadurch matchen Varianten wie ANNA MÜLLER, Anna Müller oder ein Zeilenumbruch zwischen Vor- und Nachname zuverlässiger.

Warum vollständige Namen?

Das System sucht bewusst nur nach vollständigen Namen. Ein einzelner Vorname oder Nachname reicht nicht aus. Das reduziert falsche Treffer, etwa wenn mehrere Personen denselben Vornamen haben oder ein Nachname zufällig in einem anderen Kontext vorkommt.

Wortgrenzen im Regex

Für die Suche wird ein regulärer Ausdruck mit Unicode-Buchstabengrenzen verwendet:

(?<!\p{L}) + vollständiger Name + (?!\p{L})

Dadurch muss vor und nach dem Namen entweder kein Buchstabe stehen oder ein Trennzeichen wie Leerzeichen, Komma, Punkt oder Zeilenende. Der Name wird also nicht als Teil eines längeren Wortes erkannt.

Text im PDF Ergebnis Begründung
Anna Müller MATCHED Genau ein vollständiger bekannter Mitarbeitername kommt vor.
ANNA MÜLLER MATCHED Groß-/Kleinschreibung wird durch die Normalisierung ignoriert.
Anna NO_MATCH Nur der Vorname reicht nicht für eine sichere Zuordnung.
Anna Müller und David Schneider AMBIGUOUS Mehrere bekannte vollständige Namen wurden gefunden.
Annalena Müller-Schmidt NO_MATCH für Anna Müller Der Kandidatenname wird nicht als eigenständiger Vollname gefunden.

3. Ergebniszustände des Namensmatchings

Status Bedeutung Gespeicherte Zuordnung
MATCHED Genau ein bekannter vollständiger Mitarbeitername wurde gefunden. Die passende employeeId wird als Vorschlag in der Analyse gespeichert.
NO_MATCH Kein vollständiger bekannter Mitarbeitername wurde gefunden. Es wird keine Person vorgeschlagen.
AMBIGUOUS Mehrere bekannte vollständige Namen wurden gefunden. Es wird keine automatische Person ausgewählt; die Kandidaten werden als Hinweis gespeichert.
UNREADABLE Der PDF-Text konnte nicht extrahiert werden. Es findet kein Namensmatching statt.

4. Kategorie- und Ordner-Matching

Die Ordnerlogik ist im System als Dokumentkategorie modelliert. Ein Ordner in der Oberfläche entspricht also einer Kategorie wie Verträge, Gehalt oder Kündigungen. Es werden dabei keine echten Unterordner im Dateisystem angelegt; die Zuordnung entsteht über die Spalte category_id am Dokument.

Die Klassifikation ist regelbasiert. Das System sucht im normalisierten PDF-Text nach bekannten Schlüsselwörtern. Wird genau eine Dokumentart erkannt, wird eine vorhandene Kategorie vorgeschlagen oder ein neuer Kategoriename vorgeschlagen.

Regel-Code Anzeigename Schlüsselwörter
CONTRACT Verträge arbeitsvertrag, vertrag
SALARY Gehalt gehaltsabrechnung, lohnabrechnung, gehalt, lohn
REFERENCE Zeugnisse arbeitszeugnis, zeugnis
CERTIFICATE Bescheinigungen bescheinigung
TERMINATION Kündigungen kündigung, kuendigung

Wie ein Kategorie-Treffer entsteht

Der Klassifizierer normalisiert den PDF-Text und prüft dann jede Regel. Für Schlüsselwörter wird eine nachfolgende Buchstabengrenze verwendet. Dadurch matcht zum Beispiel Arbeitsvertrag, weil das Wort auf vertrag endet. Gleichzeitig soll ein Wort wie vertragen nicht als Vertrag erkannt werden.

Anzahl erkannter Regeln Ergebnis
Keine Regel Keine Kategorie wird automatisch vorgeschlagen; manuelle Prüfung.
Genau eine Regel Die passende Kategorie wird vorgeschlagen oder als neuer Kategoriename vorgeschlagen.
Mehrere Regeln Keine automatische Kategorieauswahl, weil mehrere Dokumentarten möglich sind.

5. Bestehender Ordner oder neuer Ordner?

Wenn eine Kategorie bestätigt oder neu eingegeben wird, prüft der DocumentCategoryService, ob bereits eine Kategorie mit gleichem normalisierten Anzeigenamen existiert. So sollen nahezu gleiche Ordnernamen nicht doppelt entstehen.

Für diesen Vergleich werden Namen klein geschrieben, getrimmt und wiederholte Leerzeichen zusammengefasst. Dadurch gelten zum Beispiel Kündigungen und kündigungen als gleich. Wenn eine passende Kategorie existiert, wird deren ID wiederverwendet. Wenn nicht, wird eine neue Kategorie angelegt.

Für neue Kategorien wird zusätzlich ein technischer Code erzeugt. Dabei werden Umlaute transliteriert, Sonderzeichen durch Unterstriche ersetzt und der Code in Großbuchstaben gespeichert. Aus Kündigungen wird zum Beispiel ein Code wie KUENDIGUNGEN. Falls der Code bereits existiert, wird ein Suffix wie _2 angehängt.

6. Warum die Review-Bestätigung wichtig ist

Ein automatischer Treffer ist in dieser Anwendung bewusst nur ein Vorschlag. Die eigentliche Zuordnung passiert erst, wenn der Benutzer den Review bestätigt. Das ist besonders wichtig, weil HR-Dokumente sensibel sind und eine falsche Ablage schwerer wiegt als ein zusätzlicher Prüfschritt.

Beim Bestätigen gilt:

  • Eine employeeId ist immer erforderlich.
  • Es darf entweder eine bestehende categoryId oder ein newCategoryName übergeben werden, aber nicht beides gleichzeitig.
  • Bereits bestätigte Dokumente können nicht erneut bestätigt werden.
  • Nach der Bestätigung erhält das Dokument den Status ASSIGNED.

7. Dateisystem vs. fachliche Ordner

Die hochgeladenen PDFs werden physisch in einem lokalen Basisverzeichnis gespeichert. Dort verwendet das System einen technischen Speicherschlüssel wie 3f1a...pdf. Dieser Speicherort ist absichtlich unabhängig vom Mitarbeitenden und von der Kategorie.

Die sichtbare Ordnerstruktur in der Anwendung entsteht dagegen aus Datenbankinformationen:

  • employee_id bestimmt, zu welchem Mitarbeitenden ein Dokument gehört.
  • category_id bestimmt, in welchem fachlichen Ordner beziehungsweise in welcher Kategorie es angezeigt wird.
  • original_filename bleibt für Anzeige und Download erhalten.
  • storage_key zeigt auf die tatsächlich gespeicherte PDF-Datei.
Zusammenfassung: Das Namensmatching beantwortet die Frage: „Welcher bekannte Mitarbeitende wird im Dokument eindeutig genannt?“ Das Ordner-Matching beantwortet die Frage: „Welche Dokumentart passt anhand bekannter Schlüsselwörter?“ Beide Ergebnisse bleiben Vorschläge, bis der Benutzer sie im Review bestätigt.
