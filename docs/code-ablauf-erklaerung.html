0. Gesprächsleitfaden: Wie ich die Lösung erklären würde

Kurzfassung für den Einstieg: Ich habe die Aufgabe als kontrollierten HR-Dokumentenfluss verstanden: Ein PDF wird hochgeladen, technisch analysiert, einem bekannten Mitarbeiter vorgeschlagen und erst nach menschlicher Prüfung endgültig abgelegt. Mir war wichtig, dass die Lösung nicht magisch wirkt, sondern erklärbar, testbar und bei Unsicherheit vorsichtig bleibt.

Meine wichtigsten Überlegungen

1. Erst die sichere Kernstrecke lösen.

Die Aufgabe dreht sich im Kern um die Frage: Kann ein Dokument einem von zehn bekannten Mitarbeitern zugeordnet werden? Deshalb habe ich zuerst Upload, Textextraktion, Namensmatching, Review und Bestätigung gebaut. Alles andere ist ergänzend.

2. Keine automatische finale Entscheidung.

Auch wenn der Matcher genau einen Namen findet, bleibt das Ergebnis ein Vorschlag. In einem HR-Kontext wäre eine falsche Ablage schlimmer als ein zusätzlicher Klick. Deshalb gibt es den Status PENDING und die Review-Seite.

3. Deterministisches Matching statt LLM-Raten.

Die Mitarbeiterliste ist bekannt und klein. Daher ist ein regelbasiertes Vollnamen-Matching zuverlässiger, günstiger, schneller und besser erklärbar als ein LLM. Ein LLM kann bei offenen Inhaltsfragen helfen, aber nicht bei einer kontrollierten Identitätsentscheidung.

4. Architektur klein halten, aber mit realistischen Grenzen.

Ich habe einen modularen Monolithen gewählt, weil er für diese Größe verständlich und transaktional einfach bleibt. Gleichzeitig gibt es Ports wie DocumentStorage und DocumentClassifier, damit Storage oder Klassifikation später austauschbar sind.

5. Transparenz über MVP-Grenzen.

Authentifizierung, OCR, Virenscan, Cloud Storage und echte Retention Policies sind bewusst nicht eingebaut. Ich würde sie im Gespräch offen als Produktionsanforderungen benennen, statt so zu tun, als wären sie im MVP versteckt gelöst.

So kann man es im Gespräch erzählen

“Ich habe die Lösung bewusst konservativ aufgebaut. Das System soll nicht einfach behaupten, ein Dokument gehöre zu jemandem, sondern zeigen, warum es diesen Vorschlag macht. Wenn genau ein vollständiger Name gefunden wird, gibt es einen Vorschlag. Wenn kein Name oder mehrere Namen gefunden werden, bleibt es ein manueller Prüffall. Dadurch kann ich jeden Zustand erklären und testen.”

“KI habe ich nicht grundsätzlich ausgeschlossen. Ich habe sie nur dort eingeordnet, wo sie fachlich Sinn ergibt: bei semantischen Kategorie-Vorschlägen. Bei der Mitarbeiterzuordnung ist die Aufgabe dagegen ein kontrollierter Abgleich gegen bekannte Stammdaten. Dafür ist deterministische Logik die bessere Wahl.”

Frage im Gespräch Antwortgedanke
Warum Spring Boot und React? Solide Standardtechnologien, gut testbar, klare REST-Grenze, schnell erklärbar.
Warum PostgreSQL statt alles im Speicher? Dokumentstatus, Review-Status und Zuordnungen sind fachliche Daten und sollten persistent und nachvollziehbar sein.
Warum PDF-Dateien nicht in der DB? Die DB speichert Metadaten. Die Dateiablage ist über einen Storage-Port gekapselt und kann später durch S3 oder ähnliches ersetzt werden.
Warum Review trotz Match? HR-Dokumente sind sensibel. Die App soll unterstützen, nicht unbeaufsichtigt entscheiden.
Warum kein LLM für Namen? Weil die erlaubten Namen bekannt sind. Ein erklärbarer Vollnamen-Abgleich ist hier sicherer als probabilistische Interpretation.

1. Kurzüberblick

3. Backend-Ablauf: PDF-Upload

Der wichtigste Pfad beginnt im Frontend-Uploaddialog und endet im Backend mit zwei gespeicherten Datensätzen: einem Dokument in documents und einer Analyse in document_analyses.

Ablaufdiagramm Upload und Analyse
1. Datei wählen Benutzer wählt eine PDF im Upload-Dialog
2. POST /api/documents Frontend sendet FormData mit Feld file
3. Validieren Datei muss vorhanden und PDF sein
4. Speichern PDF wird über DocumentStorage abgelegt
5. Analysieren Text, Match und Kategorie werden berechnet
  1. DocumentUploadController.upload nimmt MultipartFile file entgegen und antwortet bei Erfolg mit HTTP 201.
  2. DocumentUploadService.upload validiert die Datei über DocumentUploadValidator.
  3. Die Datei wird über den Port DocumentStorage.store gespeichert. Im MVP schreibt LocalDocumentStorage lokal nach ./data/documents.
  4. Danach wird ein Document mit Status UPLOADED gespeichert. Zu diesem Zeitpunkt ist noch kein Mitarbeiter zugeordnet.
  5. DocumentAnalysisService.analyze extrahiert erneut den InputStream, analysiert den Text und speichert eine DocumentAnalysis mit ReviewStatus.PENDING.
  6. Falls nach dem Speichern der Datei ein Runtime-Fehler passiert, löscht der Service die bereits abgelegte Datei wieder. Das kompensiert, dass Dateisystem und Datenbank keine gemeinsame Transaktion bilden.

Analyse-Entscheidung

PDFBox Extraktion Text wird aus dem PDF gelesen
Erfolg? SUCCESS oder EMPTY/ENCRYPTED/CORRUPTED
Readable Path PersonMatcher und DocumentClassifier laufen
Analyse speichern Status, Evidenz, Score, Kategorie
Ergebnis Bedeutung Score im Backend
MATCHED Genau ein bekannter Mitarbeitername wurde gefunden. 1.0000
AMBIGUOUS Mehrere bekannte Mitarbeiternamen wurden gefunden. 0.5000
NO_MATCH Kein bekannter Mitarbeitername wurde gefunden. kein Score
UNREADABLE PDF ist leer, verschlüsselt oder beschädigt. kein Score

4. Namenszuordnung im Detail

Die Mitarbeiterzuordnung ist absichtlich deterministisch gebaut. Das bedeutet: Das System lässt kein Sprachmodell raten, welcher Mitarbeiter gemeint sein könnte, sondern prüft nachvollziehbar, ob der vollständige Name eines bekannten Mitarbeiters im extrahierten PDF-Text vorkommt.

Ablaufdiagramm Namensmatching
1. PDF-Text Text kommt aus PdfTextExtractor
2. Kandidaten EmployeeRepository liefert alle Mitarbeiter
3. Normalisieren Unicode, Kleinschreibung, Whitespace
4. Vollnamen suchen Regex mit Buchstaben-Grenzen
5. Status bilden MATCHED, NO_MATCH oder AMBIGUOUS

Wie PersonMatcher arbeitet

  1. DocumentAnalysisService.matchPerson lädt alle Mitarbeiter aus der Datenbank.
  2. Aus jedem Mitarbeiter wird ein MatchCandidate mit employeeId und vollständigem Namen gebaut, zum Beispiel Anna Müller.
  3. PersonMatcher.match normalisiert zuerst den extrahierten PDF-Text.
  4. Danach wird auch jeder Kandidatenname normalisiert.
  5. Für jeden vollständigen Namen wird geprüft, ob er als eigenständiger Texttreffer im Dokument vorkommt.
  6. Aus der Anzahl der Treffer entsteht das Ergebnis: kein Treffer, genau ein Treffer oder mehrere Treffer.
Normalisierung Warum sie wichtig ist Beispiel
Unicode-Normalisierung mit NFKC PDF-Extraktion kann Sonderzeichen unterschiedlich kodieren. Der Vergleich soll trotzdem stabil bleiben. Müller bleibt auch dann vergleichbar, wenn Zeichen intern anders repräsentiert sind.
Kleinschreibung mit Locale.ROOT Groß- und Kleinschreibung darf keine Rolle spielen. ANNA MÜLLER matcht Anna Müller.
Whitespace wird zusammengezogen PDFs enthalten oft Zeilenumbrüche, Tabs oder mehrere Leerzeichen. Anna\n\nMüller wird zu anna müller.
Regex mit Buchstaben-Grenzen Teiltreffer sollen verhindert werden. Anna Müller matcht nicht in Annalena Müller-Schmidt.

Entscheidungsfälle

Dokumentinhalt Ergebnis Begründung
Arbeitsvertrag für Anna Müller MATCHED Genau ein vollständiger Mitarbeitername wurde gefunden. Die Analyse speichert die passende employeeId als Vorschlag.
Anna hat unterschrieben NO_MATCH Nur der Vorname reicht nicht. Das verhindert falsche Zuordnungen bei häufigen Vornamen.
Zeugen: Anna Müller und David Schneider AMBIGUOUS Mehrere bekannte vollständige Namen wurden gefunden. Das Backend wählt dann bewusst keinen Mitarbeiter automatisch aus.
Dieses Dokument nennt keinen bekannten Namen NO_MATCH Kein Kandidatenname kommt im Text vor. Der Benutzer muss im Review manuell auswählen.
Leeres, verschlüsseltes oder defektes PDF UNREADABLE Die Textextraktion ist nicht erfolgreich. Ohne Text findet kein Matching statt.
Wichtig: Ein MATCHED ist in dieser App trotzdem noch keine finale Ablage. Es ist ein Systemvorschlag. Die eigentliche Zuordnung passiert erst, wenn der Benutzer den Prüffall bestätigt.

5. Warum kein LLM für die Namenszuordnung?

Ein LLM ist hier für die Mitarbeiterzuordnung nicht sinnvoll, weil die Aufgabe kein offenes Sprachverständnisproblem ist. Die App kennt die vollständige Liste der erlaubten Mitarbeiter bereits. Gesucht wird nicht eine freie Interpretation, sondern ein prüfbarer Treffer gegen genau diese Liste.

Kriterium Deterministisches Matching LLM-basierte Zuordnung
Nachvollziehbarkeit Sehr hoch: Treffer und Evidenz sind direkt erklärbar. Begründungen können plausibel klingen, sind aber nicht automatisch beweisend.
Fehlerverhalten Konservativ: kein Name oder mehrere Namen führen zu Review statt Auto-Entscheidung. Kann trotz Unsicherheit eine wahrscheinlich wirkende Antwort liefern.
Datenschutz PDF-Text bleibt lokal im Backend. Bei externer API müsste sensibler Dokumenttext an einen Anbieter übertragen werden.
Kosten und Latenz Schnell, lokal und kostenlos pro Anfrage. Zusätzliche API-Kosten, Netzwerklatenz und Ausfallrisiko.
Testbarkeit Stabil über Unit-Tests prüfbar. Antworten können modell- und promptabhängig variieren.

Der sinnvollere Einsatzpunkt für ein LLM wäre nicht die Identifikation des Mitarbeiters, sondern die optionale Dokumentklassifikation: Ein Sprachmodell könnte aus dem Inhalt ableiten, ob es zum Beispiel um Vertrag, Gehalt, Kündigung, Zertifikat oder eine neue Kategorie geht. Auch dort darf es in dieser Architektur aber nur vorschlagen. Es darf keine Kategorie anlegen und kein Dokument endgültig zuordnen.

Sinnvolle Grenze zwischen Regel und LLM
Mitarbeiter finden Deterministisch, weil erlaubte Namen bekannt sind
Kategorie vorschlagen Optional LLM-fähig, weil Inhalt semantisch interpretiert wird
Final bestätigen Immer Mensch, keine unbeaufsichtigte Entscheidung

Warum die aktuelle Lösung fachlich passt

  • Die Mitarbeiterliste ist klein, vollständig und kontrolliert.
  • Die gewünschte Zuordnung hängt an einem harten Identifikator: dem vollständigen Namen.
  • Falsche Zuordnungen wären fachlich schlimmer als ein zusätzlicher manueller Prüfschritt.
  • Ambige Fälle werden sichtbar gemacht, statt verdeckt automatisch entschieden zu werden.
  • Die Evidenz ist konkret: Name im Dokument gefunden: 'Anna Müller'.

6. Backend-Ablauf: Review und Bestätigung

Der Review-Prozess ist bewusst nicht vollautomatisch. Auch wenn das System einen Mitarbeiter sicher findet, bleibt die Analyse zuerst ein Prüffall. Erst die Bestätigung im Frontend setzt das Dokument auf ASSIGNED.

Ablaufdiagramm Prüffall bestätigen
1. Reviews laden GET /api/documents/pending-review
2. Auswahl prüfen Mitarbeiter und Kategorie im Frontend
3. Confirm senden POST /api/documents/{id}/confirmation
4. Backend validiert Employee existiert, Kategorie eindeutig
5. Zuweisen Document ASSIGNED, Analyse CONFIRMED

DocumentReviewService.confirm verhindert ungültige Zustände: employeeId ist Pflicht, categoryId und newCategoryName schließen sich gegenseitig aus, und bereits bestätigte Dokumente können nicht erneut bestätigt werden.

Ein vorgeschlagener Kategoriename wird nur dann mit Ursprung LLM_SUGGESTED gespeichert, wenn der bestätigte Name exakt dem Vorschlag entspricht. Andernfalls gilt die Kategorie als MANUAL.

10. Frontend-Abläufe

Dashboard

  1. DashboardPage ruft useEmployees auf.
  2. Die Daten werden über fetchEmployees von GET /api/employees geladen.
  3. Die globale Suche filtert lokal nach Name, Personalnummer und Abteilung.
  4. EmployeeDataGrid zeigt die Mitarbeiter und navigiert zur Detailseite.

Mitarbeiterdetail

  1. EmployeeDetailPage liest employeeId aus der Route.
  2. Parallel werden Mitarbeiterdaten, Kategorien und Dokumente geladen.
  3. Dokumente werden nach categoryId gruppiert.
  4. CategoryFolderTabs zeigt Ordner, DocumentList zeigt die passenden Dokumente.

Upload

GlobalUploadButton öffnet UploadDialog
FileDropZone PDF auswählen oder droppen
useUploadDocument Mutation mit FormData
Backend Analyse Dokument wird PENDING
Cache invalidieren pending-reviews wird neu geladen

Review-Seite

  1. ReviewPage lädt Prüfung, Mitarbeiter und Kategorien.
  2. Jeder Prüffall wird als PendingReviewCard dargestellt.
  3. Der Benutzer wählt oder korrigiert Mitarbeiter und Kategorie.
  4. useConfirmMatch sendet die Bestätigung an das Backend.
  5. Nach Erfolg werden relevante Query-Caches aktualisiert, sodass Dashboard, Detailseite und Review konsistent bleiben.

11. Fehlerbehandlung und Sicherheit

  • Das Frontend kapselt fehlerhafte HTTP-Antworten in ApiError und zeigt wiederverwendbare Fehlerzustände über ErrorState.
  • Spring Boot gibt Problem-Details über den globalen Exception Handler zurück.
  • PDF-Text, personenbezogene Inhalte und API-Schlüssel sollen nicht geloggt werden.
  • Das MVP enthält keine Authentifizierung und keine OCR-Verarbeitung. Bildbasierte oder verschlüsselte PDFs werden daher als UNREADABLE behandelt.
  • Die automatische Analyse trifft keine endgültige Zuordnung. Die Bestätigung bleibt ein expliziter Benutzer-Schritt.

13. Wichtige Grenzen des MVP

  • Keine Authentifizierung, Rollen oder Berechtigungen.
  • Keine OCR für gescannte PDFs.
  • Keine Cloud-Ablage, kein Virenscan und keine produktive Retention Policy.
  • Keine automatische finale Zuordnung ohne menschliche Bestätigung.
  • Die Klassifikation ist regelbasiert und bewusst deterministisch; ein echter LLM-Adapter ist nur architektonisch vorbereitet.
